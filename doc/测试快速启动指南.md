# 论坛项目测试快速启动指南

> 最后更新: 2026-05-04 | 作者: arfa

## 一、前置条件

### 1.1 Docker 基础设施

测试依赖以下 4 个 Docker 容器，使用 `docker-compose.yml` 一键启动：

| 容器 | 镜像 | 宿主机端口 | 容器内端口 |
|:---|:---|:---|:---|
| forum-mysql | mysql:8.0 | 3316 | 3306 |
| forum-redis | redis:7.0-alpine | 6399 | 6379 |
| forum-zookeeper | confluentinc/cp-zookeeper:7.3.0 | 2181 | 2181 |
| forum-kafka | confluentinc/cp-kafka:7.3.0 | 9092, **29092** | 9092, 29092 |
| forum-minio | minio/minio | 9000, 9001 | 9000, 9001 |

> **关键修复**: 向 Kafka 添加了 `29092:29092` 端口映射，使宿主机能通过 `localhost:29092` 连接到 Kafka（使用 `PLAINTEXT_HOST` 监听器）。

### 1.2 Go 依赖

```bash
# 核心测试依赖
go get github.com/DATA-DOG/go-sqlmock          # MySQL mock
go get github.com/alicebob/miniredis/v2        # Redis mock
go get github.com/stretchr/testify             # 断言库
```

## 二、启动流程

### Step 1: 启动 Docker 基础设施

```bash
# 进入项目目录
cd F:\Coding\Go\toOffer\Forum

# 启动所有依赖容器
docker-compose up -d mysql redis zookeeper kafka minio

# 验证容器状态
docker ps --filter name=forum
```

### Step 2: 等待 Kafka 就绪

Kafka 启动较慢（约 10-15 秒），需等待 controller 选举完成：

```bash
# 检查 Kafka 日志
docker logs forum-kafka --tail 10

# 看到类似输出表示就绪：
# [Controller id=1] Processing automatic preferred replica leader election
```

### Step 3: 预创建 Kafka Topic（可选）

由于部分测试使用 `AllowAutoTopicCreation: true`，Topic 会自动创建。但首次写入可能因 Leader 选举失败，建议预创建：

```bash
# 创建测试用 Topic
docker exec forum-kafka kafka-topics --create \
  --if-not-exists \
  --topic forum-test-votes \
  --bootstrap-server localhost:9092 \
  --partitions 1 --replication-factor 1

docker exec forum-kafka kafka-topics --create \
  --if-not-exists \
  --topic forum-test-events \
  --bootstrap-server localhost:9092 \
  --partitions 1 --replication-factor 1
```

## 三、运行测试

### 3.1 全量测试

```bash
# 运行所有测试，输出覆盖率
go test ./... -cover -timeout 90s
```

### 3.2 按模块测试

```bash
# DAO 层（纯 mock，最快）
go test -v ./internal/dao/mysql/ -cover
go test -v ./internal/dao/redis/ -cover

# Service 层（部分依赖 Docker Kafka）
go test -v ./internal/service/ -timeout 30s -cover

# Handler 层（纯 mock）
go test -v ./internal/handler/ -cover

# 中间件（纯 mock）
go test -v ./internal/middleware/ -cover

# pkg 工具包
go test -v ./internal/pkg/jwt/ -cover
go test -v ./internal/pkg/ratelimit/ -cover
go test -v ./internal/pkg/kafka/ -timeout 30s -cover    # 依赖 Docker Kafka
go test -v ./internal/pkg/minio/ -timeout 30s -cover    # 依赖 Docker Minio
```

### 3.3 单个测试

```bash
# 运行特定测试函数
go test -v -run TestVoteForPost ./internal/service/

# 运行带子测试的
go test -v -run TestJWTAuthMiddleware ./internal/middleware/
```

## 四、Mock 架构

项目采用**分层 Mock 策略**：

```
测试层           Mock 方式           依赖 Docker
─────────────────────────────────────────────────
dao/mysql     → sqlmock (DB mock)       ✗
dao/redis     → miniredis (内存 Redis)  ✗
handler       → sqlmock + miniredis     ✗
middleware    → gin test mode           ✗
pkg/jwt       → 纯逻辑测试              ✗
pkg/ratelimit → 纯逻辑测试              ✗
pkg/kafka     → Docker Kafka (29092)    ✓
pkg/minio     → Docker Minio (9000)     ✓
service/vote  → miniredis + Docker Kafka ✓ (Kafka)
service/*     → sqlmock + miniredis      ✗
```

### 关键设计决策

1. **Kafka 不使用 mock**: 在 `internal/pkg/kafka/producer.go` 中提供 `SetWriter()` 函数，测试中注入指向 Docker 的真实 Writer。
2. **Redis 使用 miniredis**: `internal/dao/redis/redis.go` 中的 `SetRDB()` 和 `InitResilienceForTest()` 用于注入内存 Redis 实例。
3. **MySQL 使用 sqlmock**: `internal/dao/mysql/mysql.go` 中的 `SetDB()` 用于注入模拟数据库连接。

## 五、常见问题

### Q1: Kafka 写入报错 code=5
**原因**: Topic 刚创建，Leader 选举未完成。
**解决**: 预创建 Topic 或等待 2-3 秒后重试。

### Q2: Kafka 启动立即退出
**原因**: ZooKeeper 中残留旧 Broker 的临时节点。
**解决**: 重启 Zookeeper: `docker-compose restart zookeeper && docker-compose up -d kafka`

### Q3: sqlmock 时间字段报错
**原因**: string 无法 Scan 进 `*time.Time`。
**解决**: 在 `sqlmock.NewRows` 中使用 `time.Now()` 而非字符串 `"2024-01-01"`。

### Q4: `encryptPassword` 哈希值不匹配
**原因**: 原实现 `h.Write(secret); h.Sum(password)` 只加盐未加密码数据。
**修复**: 改为 `h.Write(secret); h.Write(password); h.Sum(nil)`。
